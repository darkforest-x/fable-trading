# 新鲜度阈值不是拍出来的,是从管道时序算出来的

- **问题**:实盘要"实时检出",但三处新鲜度门(执行器/TG/裁决)被先后设成
  55 → 20 分钟,两个值都错——55 放进了非 tip 的迟到检出,20 则结构性挡死一切。
- **死胡同**:把阈值当"松紧偏好"来调。20 分钟听起来像"只要 tip 立刻开火",
  但没人算过管道最快能多快:信号 bar 开盘算龄,收盘要 +15min,扫描只认已收盘
  bar,入场 bar 还得再等一根,脉冲对齐 :01/:16/:31/:46,344 币扫描再吃 ~7min。
  旧管道最快入账 = 信号后 31~37 分钟——20 分钟的门下**任何信号都永远过不了**,
  且这种"静默饿死"不报错,只是永远 0 单。
- **有效路径**:先写出延迟预算表(bar 时长 15 + 脉冲对齐 1 + 扫描 ≤7),再定
  阈值 = 预算 + 余量 = 30;同时把预算里最大的可削项(等入场 bar 收盘,15min)
  用"盘口 bar 当场入账 + 下一脉冲回填真实入场"消掉,让 tip 检出在信号后
  16~23 分钟落账,30 的门才真放得进单。
- **通用规则**:任何"年龄/新鲜度/超时"阈值,先列出它守门的事件链每一环的
  最坏耗时,阈值必须 > 链路最坏和,否则就是隐形 kill switch。改阈值的 PR
  必须附这张预算表。
- **牵连**:`src/execution/config.py` max_signal_age_min、
  `src/judgment/forward.py` TG 过滤、`src/webapp/forward_payloads.py`
  FRESH_DETECT_MIN(三处必须同值);脉冲对齐见 fable-forward.timer。
